背景与问题

AI 辅助开发的现状与局限

代码生成速度的提升，并不直接等同于需求理解准确度、系统设计一致性和工程可维护性的提升。

🔄

信息传递衰减与失真

从产品需求到技术实现的多层翻译过程中，原始意图容易偏离。团队需要大量反复沟通来对齐理解。

🧩

上下文碎片化

AI 缺乏持续稳定的工程上下文，每个会话独立理解任务，无法感知上游决策、业务背景和架构约束。

🔗

需求·规范·测试 脱节

需求、规范、测试、实现之间缺少显性关联，变更影响难以快速追踪，规约无法沉淀为长期资产。

核心思路

以 Spec 为轴的 AI 协同工程

利用 AI 提升各环节执行效率，同时以结构化 Spec 将需求、规范、验证、实现串联，使 AI 和人在一致的上下文中协作。

01

需求质量前置

利用 AI 辅助的结构化追问与校验，在需求进入开发前澄清边界条件、异常路径与验收标准，将缺陷拦截在产品阶段。

02

上下文连续性

通过统一的 feature_id 将需求、接口规范、测试用例、实现代码显性关联，使 AI 在任何步骤都能获取上游结构化信息。

03

人机边界清晰

人负责决策（目标、约束、策略），AI 负责生成与执行（结构化、草稿、代码）。所有 AI 产出均需人工确认。

三层 Spec 之间为单向驱动关系：上层产出作为下层输入，形成信息密度高、传递损耗低的工程链路。

Tide · 1/2 需求讨论与发布

产品经理如何工作

口述需求 → BMAD 讨论 → PRD → 知识库推送 → Jira 创建，让每个需求有据可查。

使用流程

1
口述需求 → /tide-discuss 自动激活
产品经理在 Claude Code 中用自然语言描述需求，/tide-discuss 自动激活 BMAD 多角色讨论流。
2
BMAD 多角色讨论
AI 依次扮演分析师（Mary）、产品经理（John）等角色，从不同视角追问边界条件、异常路径、验收标准。
3
PRD 生成 + 知识库推送
讨论产出自动整理为格式规范的 PRD，推送至飞书知识库归档，团队随时查阅。
4
Jira Issue 创建
PRD 自动拆分为 Jira Issue（含 Story、优先级），关联 Figma 设计稿，立即可供开发领取。

💡 工作场景

产品经理口述需求："给后台加个企业微信登录"。

Tide 自动激活 BMAD 工作流，依次以不同角色追问：

  • 分析师 Mary → 澄清 OAuth vs 扫码、回调域配置
  • PM John → 确认多组织绑定策略、存量账号迁移
  • 架构师 → 讨论技术约束和系统集成方案

讨论完成后自动输出 PRD，推送到飞书知识库，并在 Jira 中创建开发任务（关联 Figma 原型）。

耗时：从 1-2 天人工对齐 → 30 分钟结构化讨论
关联：Figma / 飞书知识库 / Jira

Tide · 2/2 需求管理与归档

需求全生命周期管理

每个 session 贯穿 active → published → completed 全生命周期，可恢复、可追溯、可变更。

管理流程

1
入口查询与已有需求感知
再次 /tide 进入入口列表，按状态展示所有活跃会话，让用户感知已有需求上下文。支持按 sessionId、featureId、关键词搜索 sessions/ 和 archive/，关联的历史需求一目了然。
2
变更需求：关联追溯
选择「变更需求」或「重来」时，自动加载旧会话 JSON 作为参考上下文，在新需求的 BMAD 讨论中提供历史背景。旧会话标记为 superseded 并建立 parent 关联，变更脉络完整可查。
3
发布异常处理
PRD 推送到知识库或创建工单失败时，session 进入 publish_error 状态，可随时重试失败步骤。PRD Markdown 文件误删时自动从 JSON 快照恢复。
4
自动归档
全部完成（published + Jira 工单创建成功）后自动归档，从入口列表移除。归档会话可通过精确查询回溯。每次入口扫描自动搬移残留的已完结文件。

💡 状态流转示例

一个需求的完整生命周期：

# 新建讨论
🔴 active 讨论中 Mary(☑️) John(☑️) architect(⬜)

# 讨论完成 → PRD 就绪
🟢 prd_ready 待推送知识库

# 推送知识库 → 创建 Jira 工单
🟢 published 已发布 📄 飞书知识库

# 全部完成 → 自动归档
✅ completed 已归档 📎 Jira: AUTH-42

变更追溯：需求调整时旧需求 superseded，新需求 parent 指向旧记录。
/tide AUTH-LOGIN-WECOM 可搜索到全部版本，完整变更历史一目了然。

Reef · 1/4 启动与设计

从需求到设计：只需 /reef-start

一个入口 /reef-start，自动路由 Issue 或自由讨论，产出完整 Technical Spec + 实现计划。

启动流程

1
单入口 /reef-start
提供 Jira Issue 链接（/reef-start https://jira.../AUTH-42）或直接口述需求，一个命令启动所有流程。
2
自动路由与上下文获取
有 Issue → 自动拉取 PRD、Figma 设计稿和关联文档。无 Issue → 进入 OpenSpec 自由讨论，AI 按框架引导需求澄清。开发者只需按提示回答问题。
3
SDD 文档生成 + grill-me 审查
Reef 内部调用 OpenSpec 产出：方案提案（proposal）→ 技术设计（design）→ 接口规范 → 实现任务清单（tasks）。每步产出经 grill-me 多轮追问审查后进入下一阶段，最后经 spec-hardener 四道筛（含反向 grill）加固。
4
代码框架就绪
按项目配置（前端 Angular/React/Vue，后端 Java/Python）自动生成对应风格的代码骨架，开发者可直接进入 TDD 实现。

💡 工作场景

开发者启动任务 AUTH-LOGIN-WECOM，只需一步：

/reef-start https://jira.example.com/AUTH-42

Reef 自动处理后续一切：

  • ✅ 拉取 Issue 详情 + PRD 文档
  • ✅ 读取 Figma 设计稿关联
  • ✅ 创建 feature 分支
  • ✅ SDD 生成（grill-me 逐阶段审查）
  • ✅ spec-hardener 四道筛加固

开发者无需记忆 /opsx:new、/reef-gen-frontend、/reef-style-backend 等子命令——
Reef 按需在后台自动调用。所有操作在同一个引导会话中完成。

Reef · 2/4 代码规范与生成

规范先行：多维 code-style + 骨架生成

多维可组合的 code-style 体系，setup 选技术栈即生成规范文档和项目骨架，TDD 全程在规范框架内进行。

规范与生成流程

1
多维 code-style 规范
reef-style-frontend / reef-style-backend 在 setup 时按技术栈组合生成规范文档。每个维度独立产出 markdown 片段，互不干扰。维度包括框架（Angular/React/Vue）、TS 配置、CSS 方案、测试框架等。
2
代码骨架一键生成
reef-gen-frontend / reef-gen-backend 按规范文档自动创建组件、服务、路由的骨架代码，确保文件结构、命名约定、导入风格与团队规范一致。
3
TDD 实现循环
按 RED → GREEN → REFACTOR 逐 task 实现。AI 依据 code-style 规范自动编写代码，每完成一个 task 经 grill-me 审查确认后自动引导至下一个。
4
规范即约束
代码规范不只是参考文档——AI 在生成代码时主动遵守规范约束。分支范围验证和代码审查也基于同一套规范体系进行判定，确保团队一致性。

💡 技术栈组合示例

setup 时的技术栈选择决定 code-style 维度：

# 选中 Angular + Tailwind + Jest 后，code-style 自动组装
📐 framework/angular/quick-reference.md ← 组件规范
📐 ts-config/strict/quick-reference.md ← 类型约束
📐 css/tailwind/quick-reference.md ← 样式规范
📐 test/jest/quick-reference.md ← 测试规范

前端代码生成时自动遵循：Angular standalone 组件 + Tailwind class + strict TS + Jest 测试，全部自洽。

可扩展：新增维度只需添加 fragment 目录 + 更新 wizard.json，不影响已有维度。目前支持 Angular / React / Vue / Java / Python 组合。

Reef · 3/4 测试与质量门禁

四维质量防线

测试用例 → 分支验证 → 代码审查 → 设计加固，四道防线确保代码质量和业务一致性。

质量门禁流程

1
结构化测试用例生成
reef-testcase 从 Issue 描述和 PRD 上下文中提取测试场景，覆盖四维：正常流程、边界条件、异常场景、验收标准。用例既是测试指南，也帮助开发者和 AI 想清边缘情况。
2
分支范围验证
reef-scope 通过 AI 语义分析 diff 内容，自动识别变更涉及的业务领域。单领域 commit 正常通过，多领域变更阻断并给出拆分建议。支持 pre-commit 和 CI 两种门禁模式。
3
多维度代码审查
reef-review 从正确性、性能、安全、可维护性等维度进行审查，发现的问题按严重程度分级报告。子 Agent 分别审查不同维度，汇总后给出改进建议。
4
设计加固（spec-hardener）
reef-harden 在实现完成后再次审视设计合理性。第五道是"反向 grill"——主动追问"三个月后可能在哪五处后悔"，确保设计经得起时间考验。

💡 分支验证示例

git commit 被 reef-scope 阻断并自动拆分：

$ git commit -m "feat: order and payment changes"
🚫 阻断！检测到 order + payment 两个领域

# 自动生成拆分方案
→ 分支 feat/order-update 创建完毕
→ 分支 feat/payment-update 创建完毕

CI 集成：reef-scope-ci.sh 可在 GitHub Actions / GitLab CI 中运行，
PR 级别阻断多领域变更，形成团队共识。

Reef · 4/4 提交与交付

智能提交与一键 PR

reef-commit 规范提交 → reef-pr PR 描述 + Issue 关联，审阅后一键推送。

交付流程

1
智能提交信息生成
reef-commit 根据当前变更内容、OpenSpec 任务定义以及仓库历史 PR 风格，生成规范的提交信息。用户说「提交代码」「commit」自动触发，无需手动编写。
2
PR 自动创建与关联
reef-pr 读取 commits 和 OpenSpec 上下文，自动生成 PR 标题和完整描述（变更背景、影响范围、测试说明），自动关联 Jira Issue。
3
审阅后一键推送
PR 内容生成后展示给开发者审阅，可修改补充。确认无误后调用 gh 推送，做到 AI 草拟 + 人决策。
4
可追溯的交付链路
从 Issue → OpenSpec tasks → commit message → PR，整个链路可追溯。PR 描述中包含 OpenSpec change 链接，reviewer 可回溯查看完整设计上下文。

💡 提交体验

完成实现后，开发者只需说"提交代码"：

# 开发者输入
提交代码

# reef-commit 自动生成
✍️ 实现企业微信扫码登录 OAuth 流程

# 再输入"提 PR"
📤 reef-pr 自动创建 PR
📎 关联 Jira: AUTH-42
📝 描述：变更背景 + 影响范围 + 测试说明
🎉 PR ready for review!

关键原则：AI 生成草稿 → 人审阅确认 → 自动推送。
commit message 风格继承仓库历史，保持团队一致。

Sweep · 质量检测

E2E 测试如何执行

Playwright E2E 测试：初始化 → 生成测试意图 → 混合执行（原生 Playwright + AI 自愈），覆盖端到端流程。

使用流程

1
/sweep-init — 初始化项目
创建测试项目骨架，生成框架配置（Playwright）、目录结构、环境变量，标志项目可被后续流程使用。
2
/sweep-plan — 生成测试意图文档
从 Issue 或 PRD 提取验收标准，交互式覆盖功能流程、边界条件和异常场景，输出 .flow.md 测试意图文档。
3
/sweep-run — 混合执行
默认模式：预编译 .flow.spec.ts → 原生 Playwright 执行（~20x 加速）。失败时 AI 自动诊断并自愈重试。
