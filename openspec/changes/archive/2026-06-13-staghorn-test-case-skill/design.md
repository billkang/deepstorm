## Context

Reef 是 DeepStorm 开发侧套件，提供代码生成、架构合规等能力，并集成了 superpowers 用于辅助代码实现。

当前开发工作流：
1. Tide 产出 PRD → 创建 Jira Issue
2. 开发者从 Jira Issue 开始开发（Reef）
3. PRD 上传至钉钉云文档，可通过 MCP 服务获取
4. Reef + superpowers 生成代码和单元测试

目前测试用例生成缺少结构化工具支撑，测试人员产出与开发脱节，无法有效驱动单元测试质量提升。

## Goals / Non-Goals

**Goals:**
- 在 Reef 中新增 `test-case-generation` skill
- skill 输入：Jira Issue 功能描述 + PRD 上下文（通过 MCP 获取）
- skill 输出：结构化测试用例清单
- 测试用例覆盖：正常流程、边界条件、异常场景、验收标准
- 测试用例可作为 superpowers 的输入，辅助生成更完备的单元测试

**Non-Goals:**
- 不涉及测试执行或 CI 集成（属于 Sweep 待定范围）
- 不替换 Sweep 的测试侧定位
- 不涉及自动生成可执行测试代码（仅输出结构化用例清单）
- 不与外部测试管理系统集成

## Decisions

### 1. Skill 类型：纯 skill + references

**决策：** 采用纯 skill（SKILL.md）方式，配以 references 目录存放输入输出规范和模板。

**理由：**
- 不需要编译或 MCP 依赖
- 与 Tide 的模式一致，团队已有经验
- Reef 本身运行在 Claude Code 中，可直接通过 MCP 访问钉钉上的 PRD

**备选方案：** agent + hook 方案 — 复杂度高，当前阶段过度设计。

### 2. 测试用例输出格式：Markdown + 结构化模板

**决策：** 输出为 Markdown 格式的结构化测试用例清单，每个用例包含标准字段。

**理由：**
- Markdown 是 Claude Code 原生支持的格式，便于 AI 消费
- superpowers 可直接读取结构化内容辅助生成单元测试
- 人类开发者也可直接阅读

**备选方案：** JSON 格式 — 更结构化但可读性差，不利于人机协作。

### 3. 测试用例字段结构

每个测试用例包含：

| 字段 | 说明 |
|------|------|
| `ID` | 唯一标识，如 `TC-{feature}-{seq}` |
| `类型` | 正常流程 / 边界条件 / 异常场景 / 验收标准 |
| `前置条件` | 测试所需的前置状态 |
| `测试步骤` | 操作步骤描述 |
| `预期结果` | 期望的行为或输出 |
| `关联验收标准` | 对应 PRD/Jira 中的哪条验收标准 |

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 测试用例质量依赖 AI 模型能力 | 通过结构化模板约束输出质量，每次生成后允许人工 Review |
| Jira Issue 描述不够详细时，测试用例可能遗漏场景 | 支持从 PRD 获取补充上下文（MCP 读取钉钉文档） |
| 测试用例与代码实现容易脱节 | 设计为"开发过程中按需生成"，而非一次性批量输出 |
| 与 Sweep 的职责边界模糊 | 明确本次仅限于"测试用例生成"，不涉及执行/报告/CI 等 Sweep 待定范围 |
