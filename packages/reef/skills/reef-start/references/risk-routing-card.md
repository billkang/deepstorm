# 风险路由卡 — plan / tdd Mode 选择参考

> 在 superpowers 门禁阶段（阶段三→四之间）根据变更类型和风险级别，自动推荐 plan mode（直接实现 + 后置验证）或 tdd mode（完整 RED→GREEN→REFACTOR 循环）。

---

## 风险因子 → Mode 对照表

| 变更类型 | 示例 | 推荐 Mode | 依据 |
|---------|------|-----------|------|
| 文档修改 | `.md` 文件内容调整、注释修改 | **plan** | 无行为逻辑，直接可验证 |
| 配置文件 | `.json` / `.yaml` / `.toml` 配置项调整 | **plan** | 结构确定，验证成本低 |
| SKILL.md / 流程文档 | 流程描述修改、检查清单增删 | **plan** | 纯文档，无运行时行为 |
| 测试框架基础设施搭建 | 初始化 vitest / jest / pytest 配置 | **plan** | 脚手架性质，无业务逻辑 |
| 简单重构（测试覆盖充分） | 提取函数、重命名、移动文件 | **plan** | 已有测试覆盖（>80%）可验证 |
| 简单重构（测试覆盖不足） | 同上但测试覆盖 < 80% | **tdd** | 需先补测试再重构 |
| 新增业务逻辑 | Controller / Service / Repository 新增 | **tdd** | 行为变更需先写测试 |
| Bug 修复 | 功能缺陷修复 | **tdd** | 先写回归测试再修复 |
| 权限 / 安全变更 | 权限校验、鉴权逻辑、数据脱敏 | **tdd** | 高风险，需完整测试覆盖 |
| 资金 / 计费变更 | 金额计算、支付流程、账单生成 | **tdd** | 高风险，需完整测试覆盖，标记高优先级 |
| 幂等性 / 数据库迁移 | 数据库脚本、数据迁移、重入逻辑 | **tdd** | 幂等性需严格验证 |
| 状态机逻辑 | 状态流转、事件驱动 | **tdd** | 状态组合复杂，需全覆盖 |
| 并发 / 异步逻辑 | 线程、协程、消息队列消费 | **tdd** | 竞态条件需测试验证 |
| 不确定风险等级 | 无法明确归类 | **默认 auto** | Agent 推荐后人工确认 |

## Plan Mode 决定性特征

以下特征**同时**满足时，变更适合 plan mode：

1. ✅ **无运行时行为变更** — 不改变程序的控制流、数据流或副作用
2. ✅ **结果可目视验证** — 变更正确性可通过肉眼 / diff 确认
3. ✅ **无隐藏边界条件** — 不存在需要测试才能发现的边缘情况
4. ✅ **回滚成本低** — 出错后可以无风险回退

**缺少任何一个特征 → 推荐 tdd mode。**

## Plan vs TDD 执行策略差异表

| 维度 | Plan Mode | TDD Mode |
|------|-----------|----------|
| **实现前** | 直接实现 | 🔴 先写测试（RED） |
| **实现中** | 持续关注复杂度超预期信号 | 🟢 写最小实现代码 |
| **后置验证** | build → lint → test **必须全过** | 🔵 重构 → build → lint → test |
| **验证失败处理** | 修复后重新验证，通过才标完成 | 同左 |
| **测试编写** | 后置验证阶段不要求新测试 | 前置 RED 阶段已完成 |
| **代码审查** | 同标准 code-audit | 同标准 code-audit |
| **复杂度超预期** | 主动升级为 tdd mode | 不允许降级 |

## 模式选择规则

### 规则 1：plan → tdd 允许升级

Agent 在 plan mode 实现过程中，如发现变更复杂度超出预期（涉及多模块联动、边界条件复杂、异常路径多），**SHALL 主动暂停**，向用户声明并建议升级为 tdd mode。用户确认后方可切换。

**升级信号：**
- 实现时发现需要修改 3+ 个模块
- 发现隐藏的边界条件或并发问题
- 发现现有测试覆盖率不足且无法目视确认正确性
- 实现过程中引入了新的控制流分支

### 规则 2：tdd → plan 禁止降级

一旦 superpowers 门禁判定为 tdd mode，Agent **SHALL NOT** 以任何理由降级为 plan mode。即使实现过程中发现变更比预期简单，也必须完成完整的 RED→GREEN→REFACTOR 循环。

**常见违规场景（禁止）：**
- "这个 task 其实很简单，直接改吧" → ❌ 违规
- "测试已经覆盖了，不写新测试直接改" → ❌ 违规（对于 tdd mode task）
- "现在补测试太慢，先改后面再补" → ❌ 违规

### 规则 3：默认 auto 模式

风险路由**默认自动推荐**。Agent 在 superpowers 门禁中：
1. 根据变更类型和风险因子，输出风险判断表 + 推荐模式 + 推荐理由
2. **等待用户确认后**，方可进入阶段四

### 规则 4：冲突时以高风险为准

当变更同时涉及 plan 和 tdd 特征（如新增 API + 修改配置文件），**以最高风险特征为准**，推荐 tdd mode。

在 superpowers 门禁中输出以下内容供用户确认：

```
### 风险路由判断

| 变更特征 | 判定 |
|---------|------|
| 变更类型 | 新增用户注册 API（Controller + Service） |
| 涉及模块 | user-controller, user-service, user-repository |
| 运行时代码 | ✅ 是 |
| 边界条件 | 邮箱格式、密码强度、重复注册 |
| 推荐模式 | 🔴 **tdd mode** |

**理由：** 新增业务逻辑涉及多个模块联动（Controller/Service/Repository），
且有明确的边界条件和异常路径，需要完整的 RED→GREEN→REFACTOR 循环确保质量。

**请确认是否按 tdd mode 进入阶段四实现？**
```
